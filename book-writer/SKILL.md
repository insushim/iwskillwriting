# Book Writer Skill v3.0

> **Trigger**: `/book-writer`, "책 써줘", "소설 써줘", "글 써줘", "원고", "집필", "book", "novel", "write a book", "웹소설", "연재"
> **Description**: 1000+ 도구/프레임워크 분석 기반 최강 집필 오케스트레이터. 20종 소설 구조, 10종 캐릭터 프레임워크, 10종 웹소설 장르, Emotion Thesaurus, Vonnegut Story Shape, 8개 플랫폼 최적화, 10개 Python 분석 엔진, 마케팅 패키지 자동 생성.

## Overview

Book Writer는 전문 출판 워크플로우를 완전 자동화하는 올인원 집필 오케스트레이터입니다.

**핵심 역량:**
- 20종 소설 구조 프레임워크 (Save the Cat, Hero's Journey, Story Grid, Fichtean Curve, Virgin's Promise 등)
- 8종 비소설 구조 프레임워크
- Novelcrafter 스타일 Codex(스토리 바이블) 자동 생성
- Enneagram 기반 캐릭터 엔진 (자동 생성, 관계 매트릭스, 아크 설계)
- Context-Isolated 챕터 작성 엔진
- 5단계 전문 편집 (구조/문체/감정 아크/Deep POV/웹소설 최적화)
- 웹소설 특화 모드 (카카오페이지, 네이버 시리즈, 문피아 최적화)
- 10개 카테고리 QA 품질 검증 시스템
- DOCX/EPUB/PDF + 웹소설 플랫폼 + 오디오북 스크립트 출력
- Vonnegut Story Shape 감정 아크 분석 및 시각화
- 출판 플랫폼별 메타데이터 & 블러브 자동 생성

## Pipeline Phases

### Phase 0: 프로젝트 초기화 (`init_project.py`)

사용자 입력을 분석하여 프로젝트 디렉토리 구조를 생성합니다.

**입력:** 사용자의 책 요구사항 (제목, 장르, 대상 독자, 분량, 핵심 아이디어, 모드 선택 등)

### Phase 1.5: 캐릭터 설계 (Character Design)

Enneagram 기반 캐릭터 엔진을 활용하여 다차원적 캐릭터를 설계합니다.

#### 1.5-A: Enneagram 기반 캐릭터 유형 자동 추천

장르와 역할에 따라 최적의 Enneagram 유형을 자동 추천합니다.

**Enneagram 9유형 캐릭터 매핑:**

| 유형 | 이름 | 핵심 동기 | 핵심 두려움 | 최적 역할 |
|------|------|----------|------------|----------|
| 1 | 개혁가 (Reformer) | 올바름, 완벽 | 타락, 결함 | 정의로운 주인공, 엄격한 멘토 |
| 2 | 조력가 (Helper) | 사랑받기 | 사랑받지 못함 | 충성스러운 조력자, 희생적 연인 |
| 3 | 성취가 (Achiever) | 가치 입증 | 무가치함 | 야심찬 주인공, 라이벌 |
| 4 | 개인주의자 (Individualist) | 정체성 확립 | 평범함 | 예술가, 아웃사이더 주인공 |
| 5 | 탐구가 (Investigator) | 이해, 지식 | 무능함 | 탐정, 학자, 전략가 |
| 6 | 충성가 (Loyalist) | 안전, 지지 | 불안, 배신 | 충직한 동료, 의심 많은 탐정 |
| 7 | 열정가 (Enthusiast) | 자유, 행복 | 고통, 구속 | 트릭스터, 활기찬 동료 |
| 8 | 도전가 (Challenger) | 통제, 보호 | 취약함, 통제당함 | 강한 리더, 안티히어로, 빌런 |
| 9 | 평화주의자 (Peacemaker) | 조화, 평화 | 갈등, 단절 | 중재자, 평범한 영웅 |

**자동 추천 규칙:**
- 주인공-적대자 Enneagram 대립축 자동 설정 (예: 1 vs 8, 3 vs 9)
- 조력자/멘토는 주인공의 통합 방향 유형 추천
- 러브 인터레스트는 상보적 유형 추천

**Enneagram 통합/분열 방향:**
```
1->7(통합)/4(분열)   2->4(통합)/8(분열)   3->6(통합)/9(분열)
4->1(통합)/2(분열)   5->8(통합)/7(분열)   6->9(통합)/3(분열)
7->5(통합)/1(분열)   8->2(통합)/5(분열)   9->3(통합)/6(분열)

-> Positive Arc: 분열 상태에서 시작 -> 통합 방향으로 성장
-> Negative Arc: 건강 상태에서 시작 -> 분열 방향으로 타락
```

#### 1.5-B: Ghost/Wound/Want/Need 4요소 설계

모든 주요 캐릭터에 대해 4요소를 설계합니다:

```
Ghost (과거 상처 사건):
  -> 캐릭터의 현재 행동을 지배하는 과거의 트라우마적 사건
  -> 예: "어린 시절 부모에게 버림받은 경험"

Wound (내면 상처):
  -> Ghost로 인해 형성된 감정적/심리적 상처
  -> 예: "누구도 자신을 진심으로 사랑하지 않을 것이라는 확신"

Want (외적 목표):
  -> 캐릭터가 의식적으로 추구하는 구체적 목표
  -> 예: "최고의 검사가 되어 사회적 인정을 받겠다"

Need (내적 필요):
  -> 캐릭터가 진정으로 필요하지만 인식하지 못하는 것
  -> 예: "있는 그대로의 자신을 받아들이는 자기 수용"
```

#### 1.5-C: Lie/Truth 캐릭터 아크 설계

캐릭터 아크를 거짓 믿음(Lie)에서 진실(Truth)로의 여정으로 설계합니다:

**Positive Arc (긍정적 변화):**
```
Lie (거짓 믿음) -> 도전 -> 저항 -> 각성의 순간 -> Truth (진실) 수용
예: "힘만이 나를 지킨다" -> 시련과 동료 -> "진정한 강함은 타인을 신뢰하는 것"
```

**Flat Arc (신념 유지):**
```
Truth 보유 -> 세계가 Lie를 강요 -> Truth로 세계를 변화시킴
예: 주인공이 처음부터 진실을 알고, 타락한 세계를 바꿈
```

**Negative Arc (타락/환멸):**
```
Truth 보유 -> 시련 -> Lie 수용 -> 파멸 또는 비극적 결말
예: 이상주의자가 현실에 부딪혀 냉소적으로 변함
```

#### 1.5-D: Chemistry Matrix (캐릭터 관계 매트릭스)

모든 주요 캐릭터 간 관계를 매트릭스로 생성합니다:

```
Chemistry Matrix 예시:
          | 주인공   | 히로인   | 멘토     | 적대자   | 조력자
주인공    | -       | 긴장감   | 존경     | 적대     | 우정
히로인    | 긴장감   | -       | 경계     | 공포     | 신뢰
멘토      | 보호    | 중립     | -       | 숙적     | 무관심
적대자    | 집착    | 이용     | 경멸     | -       | 위협
조력자    | 충성    | 호감     | 존경     | 두려움   | -

관계 유형: 긴장감, 존경, 적대, 우정, 경계, 공포, 신뢰, 보호, 중립, 숙적,
          무관심, 집착, 이용, 경멸, 위협, 충성, 호감, 두려움, 라이벌, 연대
```

#### 1.5-E: Relationship Arc 매핑

주요 관계의 변화를 챕터별로 매핑합니다:

```
[주인공 x 히로인] Relationship Arc:
Ch 1-5:   불신 ========..  -> 호기심
Ch 6-10:  호기심 ========.. -> 존중
Ch 11-15: 존중 ========..  -> 신뢰
Ch 16-20: 신뢰 ========..  -> 사랑
Ch 21-25: 시련 ====......  -> 위기 (Midpoint 반전)
Ch 26-30: 재결합 ======.... -> 확고한 유대
```

### Phase 2: 아웃라인 생성

선택된 구조에 맞춰 상세 아웃라인을 생성합니다.

**작업 내용:**
- 전체 플롯/논점 구조화
- 챕터별 요약 (500자 이내)
- 각 챕터의 목표, 갈등, 전환점 정의
- 캐릭터 아크 매핑 (소설)
- 논점 흐름 매핑 (비소설)
- 복선(Foreshadowing) 배치 계획
- 서브플롯 타임라인 통합

**장면 기법 적용:**
- Scene-Sequel Pattern (Dwight Swain): Goal->Conflict->Disaster / Reaction->Dilemma->Decision
- Try-Fail Cycles: 점진적 실패와 성장
- Yes-But/No-And: 갈등 지속 메커니즘
- Scene Types: 액션/반응/계시/전환 균형 배치

**비트시트 생성 규칙:**
- Save the Cat: 15비트 각각에 정확한 챕터 번호 매핑
- Hero's Journey: 12단계별 전환 포인트 명시

- Episodic Structure: 에피소드별 미니 아크 + 시리즈 아크 이중 설계

### Phase 2.5: 세계관 설계 (Worldbuilding Enhanced)

장르에 맞는 체계적 세계관을 설계합니다.

#### 2.5-A: Sanderson's Laws of Magic 체크리스트

마법/초능력 체계가 있는 작품에 적용합니다:

**제1법칙 (해결 능력 = 독자 이해도):**
```
[ ] 마법으로 갈등을 해결하는 장면의 마법 체계가 독자에게 충분히 설명되었는가?
[ ] Hard Magic System vs Soft Magic System 선택이 명확한가?
[ ] Hard Magic: 명확한 규칙, 대가, 한계가 정의되었는가?
[ ] Soft Magic: 신비감이 유지되며, 마법이 문제를 쉽게 해결하지 않는가?
```

**제2법칙 (한계 > 능력):**
```
[ ] 마법의 한계와 약점이 능력보다 흥미로운가?
[ ] 대가(cost)가 존재하는가? (체력, 수명, 정신력, 자원 등)
[ ] 한계가 갈등과 긴장을 만들어내는가?
```

**제3법칙 (확장 전 깊이):**
```
[ ] 새 능력 추가 전 기존 능력의 창의적 활용을 충분히 탐구했는가?
[ ] 능력 체계가 과도하게 복잡해지지 않았는가?
[ ] 각 능력이 서사적 목적에 기여하는가?
```

#### 2.5-B: 경제/정치/문화 월드빌딩 가이드

```
[경제 체계]
+-- 화폐 / 거래 시스템
+-- 자원 분배 & 희소성
+-- 계급/계층 구조
+-- 직업 & 길드 체계
+-- 무역 & 교역로

[정치 체계]
+-- 통치 구조 (왕정/공화정/부족연합/...)
+-- 권력 투쟁 & 파벌
+-- 법률 & 사법 체계
+-- 외교 관계
+-- 군사 체계

[문화 체계]
+-- 종교 & 신앙
+-- 언어 & 문자
+-- 예술 & 학문
+-- 관습 & 금기
+-- 의식주 & 건축
+-- 역사 & 신화
```

#### 2.5-C: MICE Quotient 분석

Orson Scott Card의 MICE Quotient으로 스토리 요소 비중을 분석합니다:

```
M (Milieu / 환경):   ==========..........  50%  <- 세계 탐험이 핵심
I (Idea / 아이디어):  ====................  20%  <- 질문과 답변
C (Character / 인물): ====................  20%  <- 내면 변화
E (Event / 사건):     ==..................  10%  <- 세계 위협과 복구

-> 환경 중심 판타지: M 비중 최대 -> 세계 진입~탈출로 열림/닫힘
-> 각 MICE 스레드는 열린 순서의 역순으로 닫혀야 함 (ABBA 패턴)
```

#### 2.5-D: Iceberg Theory 적용

세계관의 보여줄 부분과 숨길 부분을 설계합니다:

```
[수면 위 (작품에 직접 등장)] - 전체의 10-20%
+-- 캐릭터가 직접 경험하는 세계관 요소
+-- 스토리에 직접적으로 영향을 주는 설정

[수면 아래 (작가만 아는 배경)] - 전체의 80-90%
+-- 역사적 배경, 미사용 지역
+-- 깊이감과 일관성의 근거
+-- 필요 시 자연스럽게 드러낼 수 있는 저장소
```

### Phase 4: 챕터 작성 (`chapter_writer.py`)

Context-Isolated 방식으로 각 챕터를 작성합니다.

**Context Injection 전략:**
각 챕터 작성 시 다음 컨텍스트를 주입합니다:
1. 전체 아웃라인 요약
2. 해당 챕터의 상세 계획 (비트, 목표, 갈등, 전환점)
3. 이전 챕터 요약 (최근 2-3개)
4. 관련 Codex 엔트리 (해당 챕터에 등장하는 캐릭터, 장소 등)
5. 문체 가이드라인
6. 감정 아크 목표 (해당 챕터의 감정 곡선 위치)
7. 캐릭터 아크 진행도 (Lie->Truth 진행률)
8. 복선 체크리스트 (이 챕터에서 심어야 할 복선, 회수해야 할 복선)

**문체 기법 옵션:**
- Deep POV -- 필터 단어 없는 깊은 몰입
- Psychic Distance (John Gardner) -- 5단계 심리적 거리 조절
- Free Indirect Discourse -- 자유간접화법
- Minimalist Prose -- 미니멀리즘 산문
- Show Don't Tell -- 감각 묘사 중심

**대화 기법 옵션:**
- Subtext Methods -- 함축 기법
- Dialogue Beats -- 행동 비트
- Said-Bookism Avoidance -- 화법 표현 절제
- Pinter Pause -- 침묵의 의미

**품질 관리:**
- 챕터별 목표 글자수 관리
- POV(시점) 일관성 검증
- 시제 일관성 검증
- 이전 챕터와의 연결성 확인
- Show vs Tell 비율 모니터링
- AI 특유 패턴 사전 억제

**웹소설 모드 챕터 작성 특화:**
- 매 화 도입부 후크 (첫 3줄 내 긴장감)
- 매 화 엔딩 클리프행어 자동 생성
- 유료화 전환 챕터(30-50화) 특별 설계
- 첫 3화 몰입도 극대화 (독자 이탈 방지)

## Orchestration Logic

### 전체 파이프라인 (Full Pipeline)

사용자가 "전체 파이프라인 실행" 또는 "자동으로 실행"을 요청하면:

```
Phase 0 (초기화)
  -> Phase 1 (구조 분석)
    -> Phase 1.5 (캐릭터 설계)
      -> Phase 2 (아웃라인)
        -> Phase 2.5 (세계관 설계)
          -> Phase 3 (Codex 생성)
            -> Phase 4 (챕터 작성)
              -> Phase 4.5 (감정 아크 최적화)
                -> Phase 5-1~5-5 (5단계 편집)
                  -> Phase 6 (기본 포맷 변환)
                    -> Phase 7 (플랫폼 최적화)
                      -> Phase 8 (오디오북, Optional)
                        -> QA (최종 검증)
```

각 Phase 완료 후 결과를 사용자에게 보고하고, 다음 Phase로 진행합니다.

### 웹소설 파이프라인 (Web Novel Pipeline)

```
Phase 0 (초기화, webnovel mode)
  -> Phase 1 (구조 분석, Fichtean/Episodic 우선)
    -> Phase 1.5 (캐릭터 설계)
      -> Phase 2 (아웃라인, 시즌제 포함)
        -> Phase 2.5 (세계관 설계)
          -> Phase 3 (Codex 생성)
            -> Phase 4 (챕터 작성, 웹소설 특화)
              -> Phase 5-5 (웹소설 최적화 편집 우선)
                -> Phase 5-1~5-4 (나머지 편집)
                  -> Phase 7 (플랫폼 최적화)
                    -> QA (최종 검증)
                      -> 연재 스케줄 생성
```

### 부분 실행

사용자가 특정 Phase만 요청하면 해당 Phase만 실행합니다:

```
"아웃라인만 만들어줘"        -> Phase 1 -> 1.5 -> 2
"캐릭터 설계해줘"            -> Phase 1.5
"세계관 만들어줘"            -> Phase 2.5
"챕터 3 다시 써줘"           -> Phase 4 (특정 챕터)
"감정 아크 분석해줘"         -> Phase 4.5
"편집해줘"                   -> Phase 5-1 ~ 5-5
"Deep POV로 고쳐줘"         -> Phase 5-4
"DOCX로 변환해줘"           -> Phase 6
"카카오페이지 규격으로"      -> Phase 7-A
"블러브 써줘"                -> Phase 7-G
"오디오북 변환"              -> Phase 8
"품질 검사해줘"              -> QA
"웹소설로 전환"              -> 웹소설 모드 활성화 + 관련 Phase
```

### Context Budget 관리

**토큰 효율 전략:**
1. Reference 파일은 필요한 것만 lazy-load
2. Codex는 챕터별로 관련 엔트리만 추출하여 주입
3. 편집 리포트는 요약본만 다음 단계에 전달
4. 이전 챕터는 요약본으로 압축하여 전달
5. 대형 프로젝트는 시즌/파트 단위로 분할 처리
6. 캐릭터 매트릭스는 해당 챕터 등장 캐릭터만 필터링

## Error Handling

1. **분량 초과**: 챕터를 분할하거나 요약하여 context window 내에서 처리
2. **일관성 오류**: consistency_checker 결과를 바탕으로 자동 수정 시도, 실패 시 사용자에게 보고
3. **포맷 변환 실패**: 패키지 미설치 시 설치 안내, 폴백으로 마크다운 출력
4. **품질 미달 (QA 점수 < threshold)**: 문제 영역 리포트 후 자동 재편집 1회 시도, 여전히 미달 시 사용자에게 구체적 개선안 제시
5. **캐릭터 일관성 위반**: 자동 경고 + Codex 기반 수정 제안
6. **세계관 모순**: 마법 체계/설정 모순 검출 시 Sanderson 법칙 기반 수정 제안
7. **플랫폼 규격 위반**: 챕터 길이/포맷 자동 조정 시도
8. **웹소설 후크/클리프행어 부재**: Phase 5-5에서 자동 보강

## Examples

```
/book-writer
판타지 소설 써줘. 제목: "별의 계승자", 10만자, Save the Cat 구조로.
```

```
/book-writer
회귀한 F급 헌터가 세계 최강이 되는 웹소설 써줘.
카카오페이지 연재, 100화 분량, 주 5회 연재.
```
